Demo guide

Walk-through, key moments, what's intentionally faked, and how to answer the questions the wholesaler will ask. Read once start-to-finish before the first demo. Use the quick reference card at the bottom during the demo itself.

Demo focus: this prototype centers on Jordan, Morocco, and Egypt — three markets with low competition for Chinese inbound and high Chinese-traveler interest right now. 4 Jordan itineraries, 4 Morocco, 3 Egypt, 2 multi-country grand tours.

1Pre-demo setup

Run through this 10 minutes before the stakeholder arrives. The first hit on any route takes 5–15 seconds to compile under Turbopack — you don't want that compile lag happening in front of the audience.

Boot the app

cd D:\dev\Yarvel\dmc-aggregator
npm run dev

Wait for ✓ Ready. Open http://localhost:3001 in an Incognito Chrome window so prior session state (drafts, languages, persona) doesn't leak in.

Verify the starting state

The landing page should show Chinese display text (深度·定制·中东之旅) on a dark warm-neutral background. If you see English instead, you have stale localStorage — DevTools → Application → Storage → Clear site data → refresh.

Click through each of these once. They'll be instant on the next hit.

  2. /wholesaler/catalog → click it-001 (Jordan flagship — this is your money shot)
  3. /wholesaler/rfqs → click any RFQ
  4. /agency/browse → click it-003 (Morocco Royal Cities — the wow page)

Then clear site data once more and refresh — fresh state, all routes hot.

Check the images

Open /agency/browse and filter to Morocco, then Egypt, then Jordan. If any card is missing a hero image, the Unsplash photo ID is stale — swap that itinerary's heroImage in lib/mock/itineraries.ts for a known-good one from a sibling itinerary. Petra and Cairo IDs are verified.

Window size + monitor

Resize to 1440 × 900. The build was designed for laptop-projector resolution. Below 1280px wide the catalog detail page starts to feel cramped.

Demo doc setup: have /pitch.html open in tab 1 and / in tab 2. Open /demo-guide.html on a phone or second monitor as a live cheat-sheet. Don't switch to this guide on the projected screen.

4What to land vs what to skim

Land hard

  1. The wholesaler portal is theirs — brand, agencies, margin control.
  2. The margin breakdown card with live markup editing. Slow down here.
  3. The agency wow page + 3-step booking + co-branded voucher.
  4. The pricing & allotments explainer banners: they answer the partner's structural questions in writing, in the UI.
  5. Bilingual locale switch: one click, no flicker.

Skim past

  • Reports page (recharts) — show for 5 seconds, don't dwell.
  • DMC admin — context only, 30 seconds max.
  • Supplier schedules and statements — visit but don't edit deeply.
  • Agency wallet ledger — mention it exists, don't walk transactions.
  • Detail pages for non-marquee itineraries (anything other than it-001 and it-003).

5What's visibly fake — be ready

Honesty here prevents being blindsided. The stakeholder should know all of these. The wholesaler will figure most out anyway if they probe — own it early.

No real backend. State lives in Zustand persisted to localStorage. Page reload preserves most state (drafts, action history, wallet deltas) within a browser session. Cross-device and cross-session, nothing persists.
Wallet deductions are simulated. No Stripe, no escrow, no real payments. The decrement animation is a UI effect on top of the Zustand store. The "Top Up" dialog adds to the same in-memory balance.
Supplier API integrations don't exist. The 7 suppliers (Petra Heritage, Atlas Mountains, Sahara Caravan, etc.) are mock entities. The Supplier portal shows what they would see if integrated, not real ops data.
The booking confirmation doesn't book anything. The 3-step flow has visible state but the success screen is hardcoded after Step 3. No supplier confirmation actually goes out.
Translations are pre-curated. Every bilingual string was written by hand for the demo. The "Pending Review" badges on some itineraries are intentional — they're our story for a translation workflow that doesn't exist yet.
Photos are stock Unsplash. Verified working for Petra, Cairo, Marrakech. Other destinations occasionally 404 — check before the demo. For production, suppliers provide real on-the-ground photography.
No auth. The persona switcher is the entire identity model. Fine for demo, obviously not production.
One hardcoded wholesaler. UB Trip is the only wholesaler configured. The multi-tenant architecture is real (CSS variable theming + data-tenant attribute on the layout root), but only one tenant has values populated. Adding a second is a config change, not code.

If the partner pushes on any of these, the consistent answer is: this is a clickable prototype, not a beta. The point is to validate the product story before we build the production system. The 40-week roadmap (in the pitch deck) is when this gets built for real.

6Q&A primer

Anticipated wholesaler questions and crisp answers. Memorize the first 4.

How do my agencies get onboarded?

Point at /wholesaler/agencies → Invite dialog. Real onboarding is KYC review + wallet setup + first credit limit. The wholesaler invites, our platform handles KYC, agency is live within 48 hours.

What's your cut?

Point at the margin breakdown card. DMC takes the middle layer — typically $200–$700 per pax depending on tier. Locked at contract signing, visible to everyone. No hidden fees, no FX games.

Can I block specific suppliers?

In production yes. Today the wholesaler catalog shows every published itinerary; supply curation lives in our platform admin. Roadmap: per-wholesaler supply filters.

What if a supplier cancels last-minute?

Today we mock the SLA queue. In production: automated fallback logic + DMC escalation + force-majeure clauses in supplier contracts. Show /wholesaler/rfqs as proof we already track time-sensitive operations.

How do I pay you?

Two modes: credit (we float working capital, monthly settlement) or prepaid (top-up wallet, draw down per booking). Their choice at contract. Show /platform/wholesalers to make it concrete.

How do I sell this to my agencies?

White-label is the answer. Their agencies log into ubtrip.safasoft.com and see only UB Trip branding. The aggregator is invisible to the agency. Demonstrate by switching personas — the wholesaler brand persists across pages.

What does this cost me to start?

Defer to follow-up commercial conversation. The stakeholder sets this number, not the demo.

Why Jordan, Morocco, Egypt specifically?

UAE and Saudi are valid but more competitive — Chinese groups already flow there. JO/MA/EG have lower competition for Chinese inbound and strong cultural interest right now. We lead with these three deliberately; UAE/SA/OM are roadmap.

Can I see real bookings volume?

Not in the demo. Hint at the platform overview KPIs being multi-tenant aggregates in production.

How long until I can actually use this?

Reference the 40-week roadmap in the pitch deck. First wholesaler pilot is weeks 12–16 with limited suppliers and one downstream agency.

7After the stakeholder review

If he says ship-it-tomorrow

  • Reset localStorage in the demo browser (DevTools → Storage → Clear).
  • Verify all 8 routes warm-compile before the wholesaler arrives (same pre-warm checklist as section 1).
  • Re-verify hero photos on every itinerary. First impression matters.
  • Have the pitch deck open in tab 1 for the opening 2 minutes and the closing slide.
  • Practice the margin-breakdown click sequence cold. It's the demo's single most important moment.

If he says fix-this-first

Common asks I'd expect:

  • More agencies in the directory — 30-min mock data edit in lib/mock/agencies.ts.
  • A second wholesaler example for "see, multi-tenant works" — 30-min in lib/mock/wholesalers.ts + tenant CSS variable.
  • More polished photos — photo curation; Ahmed-curated URLs go in lib/mock/photos.ts.
  • A real PDF download instead of a toast — non-trivial; defer to "production downloads a real PDF, this is a click-through prototype."

If he wants a Loom backup

Record the full demo (15 min) once with the section-2 script. Keep it un-edited. Useful if the demo-day network drops or the wholesaler reschedules.

If he wants a leave-behind

The pitch HTML (/pitch.html) is print-friendly — File → Print → Save as PDF. Send the PDF plus the existing stakeholder deck. The live demo is not a leave-behind; if they hit it unattended after the meeting, the gaps show. Tell them you'll set up a follow-up session if they want to see it again.
